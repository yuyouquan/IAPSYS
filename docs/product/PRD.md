## 2. 项目概述

### 2.1 项目背景

公司当前独立三方应用的发布流程依赖人工线下协调，缺乏统一的流程管控和状态跟踪平台。随着应用数量和发布频次的增长，现有方式存在以下问题：

- 发布流程节点分散，缺乏统一视图，导致信息不对称；
- 审核环节依赖线下沟通，效率低下且缺少审计痕迹；
- 缺乏班车化管理机制，发布节奏不可控；
- 各角色（需求方、运营、审核方、管理层）无法实时掌握发布进度。

基于以上痛点，需要建设一套 **独立应用发布系统 (IAPSYS)**，实现从发布申请到灰度监控的全链路流程线上化管理。

### 2.2 项目目标

1. **流程标准化**：将独立三方应用发布流程拆分为 7 个标准化节点，实现全链路线上流转；
2. **班车化管理**：引入"班车"概念，以月度班车和临时班车为单位组织批量发布；
3. **效率提升**：通过待办驱动、飞书通知等机制缩短审核响应时间；
4. **全局可视**：提供多维看板（班车视角、产品视角、状态视角），满足不同角色的数据洞察需求；
5. **质量保障**：通过物料审核、内测、灰度监控等环节确保应用发布质量。

### 2.3 项目范围

本期需求范围包含两大核心模块：

| 模块 | 子模块 | 范围说明 |
|------|--------|----------|
| 工作台 | 申请列表 | 班车创建、流程单管理、详情查看 |
| 工作台 | 待办 | 待处理事项卡片、快速跳转 |
| 流程详情 | 流程单详情 | 基础信息展示、应用卡片管理 |
| 流程详情 | 单 APK 发布详情 | 7 大流程节点全链路操作 |
| 看板 | 多维看板 | 班车视角、产品视角、状态视角 |

### 2.4 不在范围内

- 应用商店的实际上架操作（由外部平台完成）；
- APK 构建与打包系统；
- 用户权限管理后台（对接现有权限系统）。

## 3. 用户角色定义

### 3.1 角色清单

| 角色编号 | 角色名称 | 角色说明 | 核心权限 |
|----------|----------|----------|----------|
| R01 | 需求方 | 发起应用发布申请的人员，通常为应用开发或产品负责人 | 创建班车、添加应用、填写通道发布申请、查看全流程状态 |
| R02 | 运营人员 | 负责通道发布审核、物料审核的运营团队成员 | 审核通道发布申请、审核物料、回退节点 |
| R03 | 管理层（老板） | 参与会签审核的高层管理人员 | 通道发布会签审核、查看看板 |
| R04 | 物料上传者 | 负责补充完善发布物料的人员 | 编辑物料信息、上传物料文件 |
| R05 | 上架操作员 | 负责应用上架操作的人员 | 执行应用上架、录入上架结果 |
| R06 | 测试人员 | 负责业务内测环节的人员 | 执行内测验证、反馈测试结果、决定回退节点 |
| R07 | 灰度监控员 | 负责灰度发布监控的人员 | 监控灰度数据、确认发布状态 |
| R08 | 协作人 | 被节点责任人添加的协助处理人员 | 拥有对应节点的编辑权限 |

## 5. 详细功能需求

---

### 5.1 工作台模块

#### 5.1.0 布局说明

- **整体布局**：左右布局。
  - **左侧**：独立三方应用发布流程申请列表（主区域）。
  - **右侧**：待办面板。
- **待办面板交互**：支持收起和展开。
  - 收起时：左侧列表占满全宽，右侧显示收起按钮（图标形式）。
  - 展开时：右侧面板滑出，左侧列表宽度自适应缩小。
  - 默认状态：展开。

---

#### 5.1.1 独立三方应用发布流程申请列表

##### 5.1.1.1 列表筛选搜索与分页

**功能描述**：展示当前用户可见的所有独立三方应用发布流程单，支持筛选搜索和分页。

**列表字段定义**：

| 序号 | 字段名称 | 字段说明 | 数据类型 | 展示规则 |
|------|----------|----------|----------|----------|
| 1 | 班车名称 | 班车标识名称 + 时间戳 | 文本 | 格式："班车名称 时间戳"，如"3月班车 20260301120000" |
| 2 | 应用状态 | 该班车下所有应用的状态统计 | 数字 + 颜色标识 | 成功(绿色) / 拒绝(红色) / 进行中(蓝色) / 总数(黑色) |
| 3 | 申请人 | 创建该班车的人员 | 文本 | 显示用户姓名 |
| 4 | 申请时间 | 班车创建时间 | 日期时间 | 格式：YYYY-MM-DD HH:mm:ss |
| 5 | 操作 | 操作按钮列 | 按钮 | "查看详情"按钮 |

**筛选条件**：

| 筛选项 | 类型 | 说明 |
|--------|------|------|
| 班车名称 | 文本输入 | 支持模糊搜索 |
| 申请人 | 下拉选择 | 支持搜索 |
| 申请时间 | 日期范围 | 起止时间 |
| 应用状态 | 下拉多选 | 成功/拒绝/进行中 |

**分页规则**：
- 默认每页 10 条。
- 支持用户修改每页条数（可选值：10、20、50、100）。
- 显示总条数和当前页码。

##### 5.1.1.2 班车创建

**功能描述**：用户点击"申请"按钮，弹出 Modal 框创建新的发布班车。

**交互流程**：

1. 用户点击列表上方的"申请"按钮。
2. 系统弹出 Modal 框，标题为"创建发布班车"。
3. Modal 内容：
   - 班车类型选择（Radio 单选）：
     - **当月班车**
     - **临时班车**

**班车命名规则**：

| 班车类型 | 命名规则 | 示例 | 约束 |
|----------|----------|------|------|
| 当月班车 | `{月份}月班车` | 3月班车、1月班车、12月班车 | 当月仅允许创建一次；若已创建，选择"当月班车"时提示"本月班车已创建"并禁用确认按钮 |
| 临时班车 | `{月份}月临时班车{当月序号(两位)}` | 3月临时班车01、3月临时班车02、1月临时班车01 | 序号从 01 开始递增，同月内不重复 |

**确认操作**：
- 点击"确认"按钮，系统创建一条独立三方应用发布流程单。
- 创建成功后 Modal 关闭，列表自动刷新，新创建的流程单出现在列表顶部。
- 点击"取消"按钮，关闭 Modal，不做任何操作。

**校验规则**：
- 当月班车：后端校验当月是否已存在，若已存在返回错误并前端提示。
- 临时班车：后端自动计算当月序号，无需用户输入。

##### 5.1.1.3 查看详情与状态跳转

**功能描述**：

- **查看详情**：点击操作栏的"查看详情"按钮，跳转至新页面展示该流程单的详情界面。
- **状态数字跳转**：点击"应用状态"列中的状态数字（成功/拒绝/进行中/总数），跳转至该流程单的详情界面，并自动筛选出对应状态的应用。

**跳转方式**：浏览器新开页面（或路由跳转保留面包屑导航）。

**URL 参数设计**：
- 查看详情：`/workflow/{flowId}/detail`
- 状态筛选跳转：`/workflow/{flowId}/detail?status={success|rejected|processing|all}`

#### 5.1.2 流程单详情模块

##### 5.1.2.0 页面布局

- **整体布局**：上下分布。
  - **上方**：基础信息区。
  - **下方**：应用列表区（卡片形式）。

##### 5.1.2.1 基础信息展示

**功能描述**：展示当前流程单的基本信息。

**展示字段**：

| 字段 | 说明 | 展示规则 |
|------|------|----------|
| 班车名称 | 班车标识 + 时间戳 | 如"3月班车 20260301120000" |
| APK 状态 | 应用状态统计 | 成功(绿色数字) / 拒绝(红色数字) / 进行中(蓝色数字) / 总数(黑色数字) |
| 申请人 | 创建人姓名 | 文本 |
| 申请时间 | 创建时间 | YYYY-MM-DD HH:mm:ss |

##### 5.1.2.2 应用卡片列表

**功能描述**：以卡片形式展示该流程单下需要发布的所有独立应用。

**卡片字段**：

| 序号 | 字段 | 说明 | 展示规则 |
|------|------|------|----------|
| 1 | 应用图标 | 应用的 icon | 图片展示，无图标时显示默认占位图 |
| 2 | 应用名称 | 应用名字 | 文本，超长截断 + tooltip |
| 3 | 应用包名 | 包名标识 | 文本，如 com.example.app |
| 4 | 应用类型 | 应用分类类型 | 文本 |
| 5 | 应用版本号 | 通道发布申请时选择的版本号 | 文本，如 v1.2.3；未填写时显示"未指定" |
| 6 | 应用流程节点 | 当前所在的流程节点名称 | 文本，如"通道发布审核" |
| 7 | 应用流程节点状态 | 当前节点的处理状态 | 进行中(蓝色) / 成功(绿色) / 拒绝(红色) |
| 8 | 操作人 | 当前节点的操作人 | 文本 |
| 9 | 拒绝原因 | 后续节点拒绝回退时的原因 | 当被后续节点拒绝回退到前置节点时显示，红色文本 |
| 10 | 创建时间 | 应用添加到班车的时间 | YYYY-MM-DD HH:mm:ss |

**卡片交互**：
- 点击卡片跳转至"单 APK 发布流程详情"页面。
- 鼠标悬浮时卡片有轻微阴影提升效果。

##### 5.1.2.3 卡片搜索与分页

**搜索**：
- 支持按应用名称、应用包名进行关键词搜索。
- 搜索框位于卡片列表上方。

**分页规则**：
- 默认布局：一行 4 个卡片，一页 8 个卡片（2 行）。
- 每页条数可切换：8、16、24、32、64。
- 显示总数和当前页码。
- 卡片区域支持响应式，窗口宽度不足时一行展示数量自动减少。

##### 5.1.2.4 添加应用

**功能描述**：在流程单详情页面，通过"添加"按钮向当前班车中添加需要发布的应用。

**交互流程**：

1. 点击卡片列表上方的"添加"按钮（位于搜索框右侧或上方工具栏）。
2. 弹出"添加应用"Modal 框。
3. Modal 内容：
   - 可添加的应用列表（数据来源：系统已接入的应用库）。
   - 列表字段：应用图标、应用名称、应用包名、应用类型。
   - 支持搜索（按应用名称、包名模糊搜索）。
   - 支持多选（Checkbox 多选模式）。
4. 点击"确认"：
   - 选中的应用以卡片形式添加到外部列表中。
   - 每个应用创建一条独立的发布流程记录。
5. 点击"取消"：关闭 Modal，不做操作。

**特殊规则**：
- **允许重复添加相同应用**：同一应用可多次添加到同一班车中。
- **版本号唯一约束**：同一应用在同一班车中的版本号不能重复。如果添加的应用已存在于当前班车且尚未指定版本号，系统允许添加；当在"通道发布申请"节点选择版本号时需校验不与同班车同包名的其他记录版本号冲突。

#### 5.1.3 单 APK 发布流程详情模块

##### 5.1.3.0 页面布局

页面分为三大区块，上中下分布：
1. **顶部**：应用基本信息。
2. **中部**：应用发布流程（7 大节点流程图）。
3. **底部**：历史操作记录。

##### 5.1.3.1 应用基本信息

**展示字段**：

| 字段 | 说明 | 展示规则 |
|------|------|----------|
| 应用图标 | 应用 icon | 图片展示 |
| 应用名称 | 应用名字 | 文本 |
| 应用包名 | 包名标识 | 文本 |
| 应用版本号 | 通道发布申请时填写 | 文本，未填写时显示"待填写" |
| 状态 | 应用整体发布状态 | 见下方状态规则 |

**状态判断规则**：

| 状态 | 条件 | 颜色 |
|------|------|------|
| 进行中 | 存在至少一个子节点状态为"进行中" | 蓝色 |
| 已完成 | 所有子节点状态均为"已完成" | 绿色 |
| 失败 | 存在至少一个子节点状态为"拒绝" | 红色 |

> 优先级：失败 > 进行中 > 已完成。即如果同时存在拒绝和进行中的节点，状态显示为"失败"。

##### 5.1.3.2 应用发布流程 - 7 大节点

**流程图展示**：

- 横向流程图，从左到右依次展示 7 个节点。
- 每个节点显示：节点名称、当前状态（颜色标识）、责任人。
- 节点之间用箭头/连线连接。
- 节点颜色规则：
  - 已完成（成功）：绿色
  - 拒绝：红色
  - 进行中：蓝色
  - 未开始：灰色

**节点点击交互**：
- 点击节点打开对应的 Modal 框。
- **已完成节点**：Modal 打开后所有表单字段只读，仅供查看。
- **进行中节点**：Modal 打开后表单字段可编辑，用户可进行操作。
- **拒绝节点**：Modal 打开后表单字段可编辑，并在顶部醒目显示后续节点拒绝回退的原因。

**责任人与协作人机制**：
- 每个节点有默认责任人（由系统根据角色分配或流程配置指定）。
- 责任人可以点击"添加协作人"按钮，通过人员选择器添加协作人。
- 协作人拥有该节点的编辑权限，与责任人权限一致。
- 协作人列表在节点 Modal 中展示。

---

##### 5.1.3.2.1 节点1：通道发布申请

**责任人**：需求方（即添加该应用到班车的人员）。

**节点 Modal 结构**：
- Modal 标题："通道发布申请"。
- 打开时默认定位到"基础信息"模块。
- 内容分为两个 Tab/区块：
  - **基础信息**
  - **所需物料**

###### 5.1.3.2.1.1 基础信息（所有字段均为必填）

###### 5.1.3.2.1.3 提交与保存逻辑

- **实时保存**：用户在 Modal 中填写的所有内容实时保存（自动保存机制，可采用防抖策略，如用户停止输入 2 秒后触发保存）。
- **确认提交**：
  - 点击"确认"按钮，系统校验基础信息所有必填字段是否已填写。
  - 校验通过后，流程流转到下一节点"通道发布审核"。
  - 系统自动发送飞书通知给通道发布审核节点的责任人。
  - Modal 关闭，节点状态更新为"已完成"（绿色），下一节点"通道发布审核"状态变为"进行中"（蓝色）。
- **校验失败**：高亮未填写的必填字段，提示用户补充。

##### 5.1.3.2.2 节点2：通道发布审核

**责任人**：运营人员 + 管理层（老板）。

**审核机制**：包含两个子审核环节，均需通过方可流转至下一节点。

| 子环节 | 审核人 | 审核方式 | 说明 |
|--------|--------|----------|------|
| 运营人员审核 | 运营人员(R02) | 单人审核 | 运营人员对发布申请的基本内容进行审核 |
| 老板会签审核 | 管理层(R03) | 会签（多人均需通过） | 管理层对发布申请进行最终确认 |

**审核表单**：

| 字段 | 类型 | 说明 |
|------|------|------|
| 审核结果 | Radio 单选 | 通过/不通过 |
| 审核意见 | 多行文本 | 选择"不通过"时必填；选择"通过"时选填 |

**流转规则**：
- **通过**：运营审核通过且老板会签全部通过后，流程流转到"物料上传"节点，发送飞书通知。
- **不通过**：
  - 任一审核环节选择"不通过"，需填写拒绝理由。
  - 流程回退到"通道发布申请"节点，该节点状态变为"拒绝"（红色）。
  - 拒绝理由在"通道发布申请"节点的 Modal 中醒目展示。
  - 发送飞书通知给需求方。

## 6. 业务流程图

### 6.1 主流程

```
[创建班车] --> [添加应用到班车] --> [通道发布申请] --> [通道发布审核] --> [物料上传] --> [物料审核] --> [应用上架] --> [业务内测] --> [灰度监控] --> [发布完成]
```

### 6.2 详细流程流转说明

```
步骤1: 需求方在工作台点击"申请"，选择班车类型，创建发布班车
        |
        v
步骤2: 在流程单详情页，需求方点击"添加"，选择需要发布的应用（可多选，可重复添加）
        |
        v
步骤3: [节点1 - 通道发布申请]
        需求方点击应用卡片，进入单APK发布流程详情
        点击"通道发布申请"节点，打开Modal，填写基础信息和所需物料
        内容实时保存，点击"确认"提交
        --> 飞书通知审核人
        |
        v
步骤4: [节点2 - 通道发布审核]
        ├── 运营人员审核
        │   ├── 通过 --> 进入老板会签审核
        │   └── 不通过 --> 填写拒绝理由 --> 回退到节点1 --> 飞书通知需求方
        │
        └── 老板会签审核（所有老板均需通过）
            ├── 全部通过 --> 进入节点3
            └── 任一不通过 --> 填写拒绝理由 --> 回退到节点1 --> 飞书通知需求方
        |
        v
步骤5: [节点3 - 物料上传]
        物料上传者在Modal中补充必填物料信息
        基础信息回填且只读，所需物料变为必填
        点击"确认"提交
        --> 飞书通知审核人
        |
        v
步骤6: [节点4 - 物料审核]
        运营人员审核物料
        ├── 通过 --> 进入节点5
        └── 不通过 --> 填写拒绝理由 --> 回退到节点3（物料上传） --> 飞书通知物料上传者
        |
        v
步骤7: [节点5 - 应用上架]
        上架操作员查看平台数据，确认上架
        ├── 确认完成 --> 进入节点6
        └── 拒绝 --> 填写原因 --> 回退
        |
        v
步骤8: [节点6 - 业务内测]
        测试人员查看内测数据，确认结果
        ├── 通过 --> 进入节点7
        └── 拒绝 --> 选择回退目标节点（节点1/节点3/节点5） --> 填写原因 --> 回退到指定节点
        |
        v
步骤9: [节点7 - 灰度监控]
        灰度监控员监控灰度数据
        ├── 确认完成 --> 整个应用发布流程完成（所有节点绿色）
        └── 拒绝 --> 填写原因 --> 选择回退节点 --> 回退
```

### 6.3 回退机制汇总

| 当前节点 | 可回退到的节点 | 触发条件 |
|----------|----------------|----------|
| 通道发布审核 | 通道发布申请 | 审核不通过 |
| 物料审核 | 物料上传 | 审核不通过 |
| 应用上架 | 上一节点 | 拒绝 |
| 业务内测 | 通道发布申请 / 物料上传 / 应用上架（可选） | 拒绝 |
| 灰度监控 | 可选回退节点 | 拒绝 |

## 10. 术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 班车 | Shuttle | 发布批次的载体，以月度或临时方式组织一批应用的发布流程。类似于软件发布中的"发布列车"（Release Train）概念 |
| 当月班车 | Monthly Shuttle | 每月固定的发布批次，一个月仅允许创建一个 |
| 临时班车 | Temporary Shuttle | 月内额外的发布批次，可创建多个，按序号递增 |
| 流程单 | Workflow Form | 一个班车对应一个流程单，包含该批次需要发布的所有应用 |
| 节点 | Node | 发布流程中的一个环节/步骤，系统定义了 7 个标准节点 |
| 通道发布申请 | Channel Publish Apply | 第 1 个流程节点，需求方提交应用发布的基础信息和物料 |
| 通道发布审核 | Channel Publish Review | 第 2 个流程节点，运营人员和管理层对发布申请进行审核 |
| 物料上传 | Material Upload | 第 3 个流程节点，补充完善应用的多语言物料信息 |
| 物料审核 | Material Review | 第 4 个流程节点，对上传的物料进行审核 |
| 应用上架 | App Publish | 第 5 个流程节点，执行应用在各平台的上架操作 |
| 业务内测 | Business Testing | 第 6 个流程节点，对上架的应用进行内部测试验证 |
| 灰度监控 | Gray Scale Monitor | 第 7 个流程节点，监控应用灰度发布的各项指标 |
| 需求方 | Applicant | 发起应用发布申请的人员，通常是应用的开发者或产品负责人 |
| 责任人 | Owner | 每个流程节点的负责人，拥有该节点的操作权限 |
| 协作人 | Collaborator | 由责任人添加的协助处理人员，拥有对应节点的编辑权限 |
| 会签 | Co-sign | 需要多个审核人全部通过才算审核通过的审核方式 |
| 回退 | Rollback | 当某个节点审核不通过时，流程退回到指定的前置节点重新处理 |
| APK | Android Package Kit | Android 应用程序包文件 |
| GP | Google Play | Google Play 应用商店 |
| PA 应用 | Pre-installed Application | 预装应用 |
| tOS | Transsion OS | 传音定制操作系统 |
| 灰度量级 | Gray Scale Level | 灰度发布时的用户覆盖数量 |
| 物料 | Material | 应用上架所需的多语言描述文案、图标、截图等资源 |
| 工作台 | Workbench | 系统主操作界面，包含申请列表和待办两大区域 |
| 看板 | Dashboard/Kanban | 多维度数据展示面板，提供班车视角、产品视角和状态视角 |
| 飞书 | Feishu/Lark | 企业协作通讯工具，用于流程通知推送 |
| SSO | Single Sign-On | 单点登录，统一身份认证 |
| RBAC | Role-Based Access Control | 基于角色的访问控制 |
